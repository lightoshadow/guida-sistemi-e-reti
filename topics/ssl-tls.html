Sezione 6 — Sicurezza delle reti

SSL/TLS e HTTPS

TLS è il protocollo che protegge quasi tutte le comunicazioni moderne: HTTPS, email sicura, VPN SSL, autenticazione 802.1X. Devi sapere come funziona l'handshake e cosa garantisce.

TLS Handshake (TLS 1.2)

L'handshake stabilisce un canale sicuro prima di trasmettere dati. Avviene in chiaro inizialmente, poi diventa tutto cifrato.

Client Server │ │ │── ClientHello ────────────────────────→ │ (versioni TLS, cipher suites, client random) │ │ │ ←──────────────────────── ServerHello ──│ (versione scelta, cipher suite, server random) │ ←──────────────────────── Certificate ──│ (certificato X.509 del server) │ ←────────────────────── ServerHelloDone─│ │ │ │ [verifica certificato contro CA] │ │ │ │── ClientKeyExchange ──────────────────→ │ (pre-master secret cifrato con chiave pubblica server) │── ChangeCipherSpec ─────────────────── →│ (da ora si usa la chiave di sessione) │── Finished ──────────────────────────→ │ (HMAC di tutto l'handshake) │ │ │ ←──────────────────── ChangeCipherSpec ─│ │ ←───────────────────────────── Finished─│ │ │ │════════ Dati applicativi cifrati ════════│

Derivazione della chiave di sessione

  1. pre-master secret: generato dal client, cifrato con la chiave pubblica del server (RSA) — o negoziato via Diffie-Hellman
  2. master secret: derivato da pre-master secret + client random + server random con PRF
  3. session keys: derivate dal master secret — chiavi AES separate per cifratura e HMAC in entrambe le direzioni

TLS 1.3 — miglioramenti

TLS 1.3 semplifica e velocizza l'handshake:

  • 1-RTT handshake (vs 2-RTT di TLS 1.2) → connessione più rapida
  • 0-RTT resumption: per sessioni riprese, i dati possono essere inviati subito (con limitazioni di sicurezza)
  • Eliminati algoritmi deboli: RSA key exchange, RC4, 3DES, MD5, SHA-1
  • Forward Secrecy obbligatoria: solo Diffie-Hellman efimero (DHE/ECDHE) → se la chiave privata viene compromessa in futuro, le sessioni passate restano sicure

HTTPS

HTTPS = HTTP sopra TLS. Il browser apre una connessione TCP sulla porta 443, esegue l'handshake TLS, poi comunica in HTTP cifrato.

Cosa garantisce HTTPS

ProprietàMeccanismo
AutenticitàIl certificato del server è firmato da una CA affidabile → sei davvero sul sito giusto
ConfidenzialitàI dati sono cifrati con AES → nessuno può leggere il traffico in transito
IntegritàHMAC su ogni record TLS → nessuno può modificare i dati senza che se ne accorga

HTTP vs HTTPS — differenze all'esame

Best practice: tutto il traffico web deve usare HTTPS. HTTP Strict Transport Security (HSTS) forza il browser a usare sempre HTTPS per un dominio.

Cipher suite — come si legge

Una cipher suite specifica gli algoritmi usati nella sessione TLS. Formato:

TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
 │    │      │         │        │   └─ algoritmo HMAC (SHA-384)
 │    │      │         └─────────── cifratura simmetrica (AES-256-GCM)
 │    │      └─────────────────── autenticazione (RSA)
 │    └────────────────────────── scambio chiavi (ECDHE = forward secrecy)
 └─────────────────────────────── protocollo (TLS)
